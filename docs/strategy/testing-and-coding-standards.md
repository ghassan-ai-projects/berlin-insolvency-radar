# Testing And Coding Standards

**Date:** 2026-06-15
**Status:** Engineering standard for MCP v0 and production evolution

---

## Purpose

This document defines the quality bar for Berlin Insolvency Radar.

The product handles legally sensitive public records and publishes editorial intelligence. The engineering standard is therefore higher than a normal internal scraper: correctness, provenance, repeatability, and auditability matter more than speed of implementation.

The standard is:

```text
Typed boundaries.
Deterministic core.
Evidence-backed outputs.
Audited writes.
Tests before trust.
```

---

## Required Tooling

Use a Python-first toolchain:

- Python 3.12+
- `uv` or another lockfile-based dependency manager
- `pyproject.toml` as the single project config
- `ruff` for linting and formatting
- `pyright` or `mypy` for static typing
- `pytest` for tests
- `pytest-cov` for coverage
- `duckdb` for local persistence
- `pydantic` for schemas
- `langgraph` for workflow orchestration
- `langchain` for structured LLM/tool integrations
- MCP Python SDK or `fastmcp` for the MCP server

The repo should expose these commands:

```bash
make format
make lint
make typecheck
make test
make test-integration
make check
```

`make check` should run formatting check, lint, typecheck, and unit tests.

## Coding Standards

### General Style

- Use explicit, boring names.
- Prefer small modules with clear ownership.
- Keep domain logic independent from MCP, CLI, LangGraph, and DuckDB.
- Do not hide behavior in prompts when code can enforce it.
- Do not introduce abstractions until there are at least two real call sites or a clear boundary.
- Comments should explain non-obvious decisions, not restate code.

### Type Safety

- All application code must use type hints.
- External inputs and outputs must use Pydantic models.
- MCP tool inputs and outputs must be typed and validated.
- LangGraph state must be represented by typed models or typed dictionaries.
- Avoid `Any`; if it is necessary, isolate it at third-party boundaries.

### Error Handling

- Raise domain-specific exceptions in services and domain modules.
- MCP tools must convert exceptions to stable error codes.
- Error responses must be actionable for agents.
- Do not swallow exceptions silently.
- Store source-run and parse errors as data, not just logs.

Recommended error envelope:

```json
{
  "ok": false,
  "data": null,
  "warnings": [],
  "errors": [
    {
      "code": "CANDIDATE_NOT_FOUND",
      "message": "No candidate exists for candidate_id=cand_123.",
      "retryable": false,
      "next_action": "Call radar_list_candidates."
    }
  ],
  "audit_id": "audit_..."
}
```

### Database Access

- All DuckDB access must go through `storage/` repository classes or query helpers.
- No ad hoc SQL in MCP tools, agents, or domain modules.
- Migrations must be versioned and repeatable.
- Writes must be serialized through app services in v0.
- Imports and acquisitions must be idempotent.
- Every state-changing transaction must write an audit event.

### LangGraph Standards

- LangGraph nodes should be small and named after business steps.
- Graph state should carry IDs and workflow metadata, not large raw records.
- Durable facts belong in DuckDB.
- Deterministic nodes should call domain modules.
- LLM nodes must return structured outputs.
- Human approval interrupts are required before publication/export gates that could create public content.

### Agent And LLM Standards

- LLM output is never evidence.
- LLM output may propose extraction, enrichment, score rationale, draft text, or risk review.
- Claims generated by agents must be linked to evidence or marked as inference.
- Prompts must live in versioned files.
- Agent outputs must include confidence and unsupported-claim fields where relevant.
- Seeded safety tests must verify that risk-review agents block unsupported or personal-data claims.

### MCP Standards

- MCP tools should be thin wrappers over application services.
- All MCP outputs must use the shared result envelope.
- State-changing tools must accept `actor` or `reviewer`.
- Import/acquisition tools must support `dry_run`.
- List tools should default to actionable queues, not raw dumps.
- Tool results should include `next_action` when it helps an agent continue safely.

## Test Strategy

The test strategy is layered. Fast deterministic tests carry most of the confidence. Integration tests prove storage and workflow behavior. Evals are added for LLM nodes once those nodes exist.

### Test Pyramid

```text
Many unit tests
Some integration tests
Focused workflow tests
Few contract tests
Small LLM eval suite
Manual acceptance before public output
```

### Unit Tests

Run on every change.

Must cover:

- scoring formula and category thresholds
- score input validation
- legal-form allowlist
- consumer/personal rejection patterns
- publication status transitions
- dedupe keys and normalization
- source trust-level rules
- free-tier field suppression
- retention policy helpers
- Markdown rendering helpers

Examples:

- `GmbH & Co. KG` is allowed.
- individual debtor text is quarantined.
- same company/date/case number dedupes.
- legal risk subtracts from score.
- unapproved candidate cannot be included in issue output.

### Integration Tests

Run before merging implementation work.

Must cover:

- DuckDB migration creates all tables.
- repository inserts and reads candidates, evidence, scores, reviews, issues, audit events.
- legacy import dry-run writes nothing.
- legacy import real run is idempotent.
- duplicate legacy records collapse into one candidate.
- review writes candidate status, score, review row, and audit event.
- issue draft persists and export writes local Markdown.
- source-run errors are stored and visible through health/audit.

Integration tests should use temporary DuckDB files under test temp directories, never the real `data/radar.duckdb`.

### LangGraph Workflow Tests

Must cover:

- import workflow happy path
- import workflow dry-run path
- import workflow duplicate path
- review workflow approve/reject/needs-more-info paths
- review workflow invalid transition
- issue draft workflow blocks unapproved candidates
- issue draft workflow blocks candidates without evidence
- workflow failures leave auditable error state

Graph tests should assert final state and database side effects.

### MCP Contract Tests

Must cover every v0 tool:

- `radar_health`
- `radar_import_legacy_scout`
- `radar_list_candidates`
- `radar_get_candidate`
- `radar_review_candidate`
- `radar_create_issue_draft`
- `radar_export_issue`
- `radar_audit_trail`

For each tool, test:

- input validation
- success envelope shape
- failure envelope shape
- audit behavior for writes
- `dry_run` behavior where applicable
- useful `next_action` in common agent workflows

### Fixture And Golden Tests

Maintain fixtures for:

- Berlin GmbH filing
- UG filing
- GmbH & Co. KG filing
- AG filing
- sole proprietor / individual debtor to quarantine
- ambiguous company requiring review
- duplicate legacy rows
- missing case number
- malformed source row
- empty source run

Golden outputs:

- normalized candidate JSON
- evidence JSON
- issue Markdown
- audit trail summary

Golden tests should be reviewed deliberately when output changes.

### LLM Evaluation Tests

Add once agent nodes exist.

Must evaluate:

- extraction returns required fields with confidence
- extraction preserves evidence snippets
- analyst does not invent revenue, employee count, or assets
- risk reviewer flags unsupported claims
- risk reviewer flags financial-advice language
- risk reviewer flags personal-data leakage
- newsletter draft includes disclaimer and source labels

Use a small curated eval set first. Do not block early deterministic implementation on large eval infrastructure.

### Compliance Tests

These are non-negotiable.

Must prove:

- consumer/personal records do not reach `publish_ready`
- first names, birthdays, and private addresses are suppressed or quarantined
- public output cannot include candidates without evidence
- free-tier output suppresses administrator contact details until legal review
- external publish/send methods are unavailable in v0
- old and new production pipelines are not both configured as active in this repo

### Performance And Reliability Tests

For v0:

- import 1,000 legacy-style rows within a reasonable local runtime
- repeated import is idempotent
- health query remains fast on expected local data volume
- export is deterministic

For official scraper phase:

- retry behavior
- timeout handling
- empty-result day
- parser failure recording
- repeated acquisition idempotency

## Coverage Targets

Coverage is not the goal, but it is a useful guardrail.

Minimum targets:

- domain modules: 95%+
- services: 85%+
- storage repositories: 80%+
- MCP tool wrappers: contract tests for every tool
- agents/prompts: eval coverage for critical behaviors, not line coverage

No coverage target can compensate for missing compliance tests.

## Definition Of Done

Phase-level acceptance gates are defined in `strategy/phase-acceptance-tests.md`. The checklist below applies to every implementation ticket.

For any implementation ticket:

- code is typed
- formatter and linter pass
- typecheck passes or exceptions are documented
- relevant unit tests exist
- integration tests exist for storage or workflow changes
- MCP contract tests exist for tool changes
- every state-changing path writes audit
- docs are updated when behavior or contracts change
- no real legacy DB is mutated during tests
- no external publish/send is introduced without explicit approval

For MCP v0 specifically:

- `radar_health` starts cleanly against a fresh DuckDB
- legacy import dry-run and real import work
- candidates are deduped
- review writes status, score, review, and audit
- issue draft/export produces Markdown
- unapproved/quarantined candidates cannot be exported
- consumer/personal records are rejected or quarantined
- `make check` passes

## Review Checklist

Use this checklist in code review:

- Is the business rule implemented in code rather than prompt text?
- Can this operation be rerun safely?
- Does this write create an audit event?
- Does the candidate have evidence for every public claim?
- Could this leak consumer/personal data?
- Does the MCP response tell an agent what to do next?
- Are DuckDB writes centralized?
- Are tests using temp databases and fixtures?
- Would this still work if OpenClaw is only a remote control?
- Does this create a second production pipeline accidentally?

## CI Gate

Before code is considered production-ready, the project should have a CI job equivalent to:

```bash
make check
make test-integration
```

The CI should fail on:

- formatting drift
- lint errors
- type errors
- unit test failures
- integration test failures
- missing or changed golden outputs unless explicitly approved

## Local Development Safety

- Default database path for development: `data/radar.duckdb`.
- Tests must never use the real development DB.
- Treat `/Users/ghassan/my-projects/insolvency-scout/data/insolvency_scout.duckdb` as production data.
- Legacy DB must be opened read-only.
- Tests should use fixture copies or temp clones, not the live legacy DB.
- Any optional smoke test against the live legacy DB must assert unchanged file size, modified time, and content hash.
- Clone legacy data only to an explicit new path such as `data/legacy_snapshots/` or a temp directory.
- Destructive cleanup commands must target temp directories only.
- Secrets belong in local environment variables or ignored config files.
- Exported drafts belong under `data/exports/`.
